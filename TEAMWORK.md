# Командная работа над snowbag — правила, чтобы не конфликтовать

Нас двое, работаем параллельно на разных компьютерах:
- **Виктория** — расчёт **Теплотехника** (ветка `ttr`)
- **Коллега** — расчёт **Снеговые мешки** (ветка, например, `bags`)

Главная мысль: конфликт возникает только когда **двое меняют одни и те же строки одного файла**. Если каждый пишет свой расчёт в своих файлах и мы часто синхронизируемся — конфликтов почти не будет.

---

## Однократная настройка (просьба к коллеге)

1. **Заведи свою ветку и перестань писать прямо в `master`.**
   В Cursor: внизу слева щёлкни по названию ветки → **Create new branch…** → назови `bags` → переключись на неё.
   (Или командой: `git switch -c bags`, затем «Publish Branch».)
   `master` — это общий «чистовик», в него попадаем только через Pull Request.

2. **Включи защиту `master`** (это может только владелец репозитория — то есть ты).
   GitHub → репозиторий **snowbag** → **Settings** → **Branches** → **Add branch ruleset** (или Add rule) →
   - Branch name pattern: `master`
   - включи **Require a pull request before merging**
   - сохранить.
   Тогда никто (включая нас) не сможет случайно записать в `master` напрямую — только через проверку.

---

## Кто какие файлы трогает (карта зон)

**Зона коллеги (мешки):**
- `api/internal/analytics/snowbags.go` и связанные расчёты мешков
- компоненты мешков во `web/src/components/` (RoofCanvas, WindRose, Yandex-карта и т.п.)

**Зона Виктории (теплотехника):**
- **новые** файлы, например `api/internal/analytics/heattech.go`
- **новые** экраны/компоненты во `web/src/` под теплотехнику

**Правило №1:** свой расчёт пишем в своих **новых** файлах, в чужие не лезем.

**Общие файлы (зона риска — трогаем осторожно, предупреждаем друг друга):**
- `web/src/router/index.ts`, `web/src/types/index.ts`, `web/src/api/client.ts`
- `api/internal/handler/handler.go`, `api/internal/analytics/engine.go`
- `package.json`, `go.mod`, `docker-compose.yml`

Для общих файлов: меняем **маленькими кусочками**, **сразу отправляем**, и **пишем друг другу** перед правкой.

---

## Дневной ритм (повторяем каждый день)

1. **Утром — забрать свежее.** Подтянуть свежий `master` в свою ветку:
   `git fetch origin` → `git merge origin/master`
   (В Cursor: меню веток → Pull/Sync; затем влить master в свою ветку.)
2. **Работаем** только в своей ветке.
3. **Сохраняем** (commit) маленькими шагами с понятным описанием, что сделал.
4. **Отправляем** (push) — желательно несколько раз в день, не копим.
5. Когда часть готова — **Pull Request** на GitHub: второй смотрит изменения и вливает в `master`.

Чем чаще синхронизируемся (п.1) и чем меньше порции (п.3–4) — тем меньше конфликтов.

---

## Если всё же случился конфликт

Это не страшно — git просто покажет в файле два варианта одного места и попросит выбрать.
В Cursor такие места подсвечиваются с кнопками «Accept Current / Accept Incoming / Accept Both».
Если непонятно — не угадывать, а списаться друг с другом: чей кусок правильный.

## Шпаргалка: запуск проекта локально

- Включить: `docker compose up -d` (из папки проекта)
- Выключить: `docker compose down`
- Приложение: http://localhost:3000
