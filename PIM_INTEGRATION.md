# Архитектура интеграции с ПИМ (QRC API) — модуль теплотехники

Цель: сейчас работаем на демо-данных, но всё готово так, что для «боевого» режима достаточно
**вставить токен** и **указать коды атрибутов** — переписывать логику не придётся.

## Факты по API (из доки QRC v1)
- База: `https://qrcai.tndev.ru/api` (prod также упоминается `https://qrc.ai/api` — **уточнить, какой для нас**).
- Авторизация: заголовок `Authorization: Bearer <токен>`. Токен запросить письмом на **pim@tn.ru** (указать название ресурса и описание).
- Лимит: **2 запроса/сек**.
- Главный запрос: `GET /v1/product/{ЕКН}?locale=ru_RU&attributes=...` — карточка продукта/системы по ЕКН.
- Доп.: `GET /v1/product/{ЕКН}/certificates`; `POST /v1/product/updated?from&to` — список обновлённых ЕКН (для инвалидации кэша).
- Данные: `groups → attribute{attribute_type, data, label, value}`. Метрики приходят как `{amount, unit}` (для λ, толщины и пр.).

## Принципы
1. **Provider-абстракция** (один интерфейс, две реализации):
   - `DemoProvider` — отдаёт демо-системы и материалы из локального JSON/БД (сейчас).
   - `QRCProvider` — реальные HTTP-запросы к ПИМ (потом).
   - Переключение по env: `PIM_MODE=demo|live`, `PIM_TOKEN=...`, `PIM_BASE_URL=...`.
2. **Кэш в БД.** ПИМ — мастер, но ответы кэшируем (скорость + лимит 2 rps). Обновление по `/product/updated`.
3. **Карта атрибутов** — отдельный конфиг: «наше поле → код атрибута ПИМ». Коды уточняем по таблице атрибутов (Google Sheet из доки). Пока — заглушки-плейсхолдеры.
4. **Rate-limit 2 rps** встроен в клиента.

## Таблицы БД (миграция 005) — в т.ч. «таблица с id»
- `pim_systems` — **карта систем кровли**: `id` (наш слаг, напр. `tn-standart`), `ekn` (ЕКН в ПИМ), `title`, `raw_json` (кэш), `updated_at`. ← это и есть таблица с id систем.
- `pim_materials` — материалы: `id`, `ekn`, `title`, `lambda` (λ), `mu` (паропроницаемость, для разд.8), `density`, `raw_json`, `updated_at`.
- `pim_system_layers` — состав системы: `system_ekn`, `material_ekn`, `order`, `default_thickness_mm`.
  (Если ПИМ отдаёт состав внутри карточки системы — храним в `raw_json` и парсим; **уточнить по таблице атрибутов**.)

## Пакет Go: `api/internal/pim/`
- `client.go` — HTTP, Bearer-токен, rate-limit 2 rps, ретраи.
- `provider.go` — интерфейс `Provider` + `DemoProvider` + `QRCProvider`.
- `types.go` — `Product`, `Attribute` (тип/значение), `Metric{Amount, Unit}`.
- `attributes.go` — карта кодов: λ, толщина, μ, плотность, состав. **Коды — заглушки до уточнения.**
- `cache.go` — кэш в БД + синк по `/product/updated`.

## Что нужно получить (менеджерские задачи Виктории)
1. **Токен** — письмо на pim@tn.ru. Ресурс: «Сервис теплотехнического/паро- расчёта кровли (snowbag)», описание — кратко.
2. **Базовый URL** — подтвердить (`qrcai.tndev.ru` или `qrc.ai`).
3. **ЕКН систем кровли ТН** — список (ТН-КРОВЛЯ Стандарт/Проф/Смарт/Балласт → их ЕКН). Заполним `pim_systems`.
4. **Коды атрибутов** из таблицы атрибутов ПИМ для: теплопроводность λ, толщина слоя, **коэф. паропроницаемости μ**, плотность, **состав системы** (как система ссылается на материалы). Доступ к Google-таблице — я разберу коды сам.

## Готовность к «боевому» режиму
Когда придут токен + ЕКН + коды атрибутов: задаём `PIM_MODE=live`, `PIM_TOKEN`, заполняем `pim_systems` реальными ЕКН и коды в `attributes.go` — расчётная логика не меняется.

## ✅ Реализовано (живая синхронизация)
- **Скрипт:** `scripts/pim_sync.py` — тянет 4 системы из ПИМ → состав (associations) →
  материалы (λ/μ/ρ) → SQL для `thermal_systems / thermal_materials / thermal_system_layers`.
- **Запуск:** `PIM_TOKEN=... python3 scripts/pim_sync.py | docker compose exec -T postgres psql -U snowbag -d snowbag`
  (токен — из `.env`, см. `.env.example`).
- **ЕКН систем:** Стандарт 10000008, Проф 10000025, Смарт 10000006, Балласт 10000013.
- **Коды атрибутов материала:** λ — `atr_term_expl_b`/`atr_term_expl_a`/`thermal_d`;
  μ — `vapor`; ρ — `atr_density`; толщина — `Thikness`; имя — `atr_name_for_website`.
  Запрос с фильтром `?attributes=...` (5 МБ → ~12 КБ).
- **Слои** распознаются динамически из `associations` (`syst_main_<роль>_<N>`), классификация по смыслу.
- Демо-сидинг в `thermal_store.go` теперь ставится только на пустой базе (реальные данные не трогает).
