# Вклад в развитие TN Life UI Kit

## Занесение изменений (Создание `Merge request`)

Для упрощения приёмки и ревью изменений в `TN Life UI Kit` (далее `UI Kit`), необходимо создавать `Merge request` (далее `MR`) после обсуждения их в соответствующем `Issue` для предварительной модерации.

## Последовательность внесения изменений

1. Создание `Issue` с обсуждением/предложением какого-то изменения.
2. Аппрув `Issue` от стороннего `Lead frontend` для взятия в работу.
3. Создание `MR` с пометкой связанного `Issue`.
4. Первичное ревью изменений `Lead frontend`.
5. `Lead frontend` оставляют замечания в формате тредов `Thread`. Способы ревью остаются за `Lead frontend`.
6. Автор `MR` исправляет замечания, отвечает на вопросы - даёт понять что можно проверить ещё раз.
7. `Lead frontend` - автор `Thread` отмечает его как завершённый (`Resolve thread`).
8. После того как минимум три `Lead frontend` пометило `MR` как проверенный, он мержится в master-ветку и формируется основной тег репозитория и автоматически формируется тег для npm.

## Написание `Issue`

Issue пишется в трёх случаях: новый функционал (Feature Request), ошибка (Bug Report) и оптимизационный вопрос/предложение:

1. Для нового функционала ожидается:
   - Для удобства можно выбрать шаблон `feature`
   - Ссылка на фигму Дизайн-системы с новым функционалом.
   - Описание компонента для доработки или полное описание нового компонента
   - Скриншоты оттуда
   - Подсказки по реализации приветствуются.
   - Метка `feature`
2. При описании ошибки следует:
   - Для удобства можно выбрать шаблон `bug`
   - Указать версию `UI Kit`.
   - Указать проблемный компонент если имеется.
   - Подробно описать ошибку:
      1. Основное текстовое описание ошибки.
      2. Код использования компонента.
      3. Шаги воспроизведение ошибки.
      4. Ожидаемое поведение.
      5. Фактическое поведение.
   - Метка `bug`.
3. Оптимизационный вопрос или предложение имеет свободную форму описания, но чем подробнее, тем лучше. Для таких `Issue` ставится метка `support`.

После обсуждения `Issue` и аппрува со стороны хотя бы одного `Lead frontend` не из вашей команды (компании) задачу можно брать в работу и в последствии `MR`.

## Правила оформления `MR`:

1. Каждому `MR`-у должен соответствовать ранее заведённый и аппрувнутый `Issue`.
2. Название ветки может быть любым.
3. Название (`Title`):
   1. Оно должно иметь понятное краткое содержание вносимых изменений.
   2. Не возбраняется обозначение задач из таск-трекера, например префикс `support/TNLIFE-1234`.
4. Описание (`Description`):
   1. Концептуально опишите реализацию и возможности вносимых изменений.
   2. Приветствуются скиншоты вёрстки компонента, демонстрирующие внесённые изменения.
   3. Допускаются листинги кода. Также, если необходимо, можно писать комментарии к фрагментам когда в соответствующем разделе `MR`.
   4. Не будет лишним ссылка на фигму Дизайн-системы.
   5. В любом месте оставьте ссылку на связанный `Issue` (`#<номер Issue>`).
5. Метки (`Labels`). В соответствии с вносимыми изменениями можете ставить соответствующие метки:
   - `component` - создание нового компонента.
   - `feature` - дополнение уже существующего компонента.
   - `fix` - исправление работы компонента или всего UI Kit.
   - `support` - прочие изменения.
6. В роли `Assignee` отмечайте самого себя, а в роли `Reviewer` - вашего `Lead frontend`. Если вы им и являетесь, то отмечайте [`@tages_alexander.nilov`](https://gitlab.tn.ru/tages_alexander.nilov).
7. Остальных `Lead frontend` стоит отмечать в конце описания.
